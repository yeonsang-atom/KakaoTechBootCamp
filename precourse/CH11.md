## CH 11. 프로세스와 스레드  

### 1. 핵심 개념 3개  

- 프로세스
- 스레드
- 멀티 프로세스 vs 멀티 스레드

---

### 2. 내 말로 설명하기  

**프로세스**는 자원을 할당받아 실행 중인 프로그램을 의미하며, CPU를 할당받아야 실제로 실행될 수 있다. 여러 개의 프로세스가 동시에 실행되는 것처럼 보이지만 실제, CPU를 번갈아 사용하면서 동시에 실행되는 것처럼 보인다.

운영체제는 각 프로세스를 관리하기 위해 **PCB(Process Control Block)**라는 자료구조를 사용하며, 여기에는 PID, 레지스터 값, 상태, 메모리 정보 등이 저장된다. 특히 컨텍스트 스위칭 시 이전 상태를 복원하기 위해 레지스터 값이 중요하다.

프로세스는 생성, 준비, 실행, 대기, 종료 상태를 가지며, 상황에 따라 상태가 전환된다. (ex, 실행 중인 프로세스는 타이머 인터럽트가 발생하면 준비 상태로 돌아가고, I/O 작업이 발생하면 대기 상태로 전환된다.)  

**스레드**는 프로세스 내부의 실행 흐름 단위로, 하나의 프로세스 내에서 여러 스레드가 동시에 실행될 수 있다. 스레드는 코드, 데이터, 힙 영역을 공유하고, 각자 프로그램 카운터와 스택을 가진다.  

**멀티 프로세스**는 여러 개의 프로세스를 실행하는 방식으로 각 프로세스가 독립된 메모리 공간을 가지기 때문에 안정성이 높다. 반면 **멀티 스레드**는 하나의 프로세스 내에서 여러 스레드를 실행하는 방식으로 자원을 공유하기 때문에 효율적이지만 동기화 문제가 발생할 수 있다.

---

### 3. 추가로 찾아본 내용 (AI 활용)  

**스레드 동기화 문제 해결 방법**  

- 여러 스레드가 공유 자원에 동시에 접근하면 데이터 불일치 문제가 발생할 수 있다.  

1. Mutex (Mutual Exclusion) - 상호배제  
- 하나의 스레드만 임계 구역에 접근할 수 있도록 lock을 거는 방식  
- 다른 스레드는 lock이 해제될 때까지 대기

2. Semaphore  
- 여러 개의 자원에 대해 접근을 제어하는 방식  
- 카운팅 값을 통해 동시에 접근 가능한 스레드 수를 제한

→ 이를 통해 멀티 스레드 환경에서 발생하는 경쟁 상태(race condition)를 방지할 수 있다.  

**경쟁 상태(race condition) : 여러 프로세스 또는 스레드가 동시에 같은 자원에 접근하여 결과가 실행 순서에 따라 달라지는 상황

---

**IPC (Inter-Process Communication)**

- 서로 다른 프로세스 간에 데이터를 주고받기 위한 방법

1. Pipe  
- 한 프로세스에서 다른 프로세스로 데이터를 한 방향으로 전달  

2. Message Queue  
- 메시지를 큐 형태로 저장하고, 프로세스가 이를 읽는 방식  

3. Shared Memory  
- 분리된 여러 프로세스가 일부 메모리 공간을 의도적으로 공유하여 빠르게 데이터 교환 (스레드의 장점을 적용하여 사용)  
- 대신 동기화 문제가 발생할 수 있음  

4. Socket  
- 네트워크를 통해 다른 프로세스와 통신하는 방식  
- 서버-클라이언트 구조에서 사용됨  

→ IPC를 통해 독립적인 프로세스 간에도 협력하여 작업을 수행할 수 있다.  

---

### 4. 개발 또는 일상과 연결해 본 예시

파일을 다운로드하면서 동시에 다른 작업을 수행할 수 있는 이유는 다운로드 작업이 별도의 스레드 또는 프로세스로 처리되기 때문이다.

이 과정에서 CPU는 여러 작업을 번갈아 수행하며 동시에 실행되는 것처럼 보이게 한다.  

웹 서버에서는 여러 사용자의 요청을 동시에 처리하기 위해 스레드 기반으로 요청을 분산 처리한다.

이때 스레드를 사용하면 자원을 효율적으로 사용할 수 있지만, 동시에 같은 데이터를 접근할 경우 동기화 문제가 발생할 수 있다는 점도 함께 고려해야 한다.

---

## 실제 개발이나 운영 상황에서는 어떤 trade-off가 있을까?

멀티 프로세스는 각 프로세스가 독립된 메모리 공간을 가지기 때문에 안정성이 높지만, 생성 비용과 메모리 사용량이 크다.

반면 멀티 스레드는 자원을 공유하기 때문에 효율적이지만, 동기화 문제가 발생할 수 있고 하나의 스레드에서 문제가 생기면 전체 프로세스에 영향을 줄 수 있다는 단점이 있다.
