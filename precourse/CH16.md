## 전체 학습 소감

파일 시스템을 학습하면서, 운영체제가 데이터를 단순히 저장하는 것을 넘어서 어떻게 구조적으로 관리하고 접근을 제어하는지 이해할 수 있었다.

파일 접근이 반드시 운영체제를 통해 이루어진다는 점에서, 운영체제가 시스템 자원의 안전성과 일관성을 유지하는 역할을 한다는 것을 다시 한 번 느낄 수 있었다.

파일 할당 방식에서는 연속 할당의 단점인 외부 단편화 문제를 시작으로, 이를 해결하기 위한 연결 할당, 색인 할당 방식을 학습할 수 있었다.

또한 FAT와 유닉스 파일 시스템(i-node)을 통해 실제 운영체제에서 파일을 어떻게 관리하는지 구조를 이해할 수 있었고, 특히 i-node 구조를 통해 큰 파일을 효율적으로 관리하는 방식이 인상적이었다.

### 2. 내 말로 설명하기

*파일 시스템*은 운영체제에서 파일과 디렉토리를 관리하는 프로그램으로, 데이터를 체계적으로 저장하고 접근할 수 있도록 도와준다. *파일*은 보조기억장치에 저장된 의미 있는 정보의 집합이며, 반드시 운영체제를 통해서만 접근할 수 있다.

*디렉토리*는 파일을 효율적으로 관리하기 위한 구조로, 파일의 위치와 정보를 저장한다. 경로를 통해 특정 파일의 위치를 지정할 수 있으며, 절대 경로와 상대 경로가 존재한다.

파일은 보조기억장치에 *블록 단위*로 저장되며, 이를 어떻게 할당하느냐에 따라 성능과 효율이 달라진다.

연속 할당은 파일을 연속된 블록에 저장하는 방식으로 구현이 간단하지만 외부 단편화가 발생할 수 있다. 이를 해결하기 위해 불연속 할당 방식이 사용된다.

불연속 할당 방식 중 *연결 할당*은 각 블록이 다음 블록을 가리키는 구조로 외부 단편화를 줄일 수 있지만, 순차 접근만 가능하여 임의의 블록에 접근하는데 속도가 느리다는 단점이 있다. 반면 *색인 할당*은 모든 블록 주소를 인덱스 블록에 저장하여 임의 접근이 더 빠르다.

FAT 파일 시스템은 연결 정보를 테이블 형태로 관리하여 접근 속도를 개선한 방식이고, 유닉스 파일 시스템은 **i-node** 구조를 사용하여 파일의 블록 주소를 효율적으로 관리한다.

### 3. 추가로 찾아본 내용 (AI 활용)

**ext2 파일 시스템**

- ext2는 리눅스에서 사용되는 전통적인 파일 시스템 중 하나이다.
- i-node 기반 구조를 사용하여 파일의 메타데이터와 데이터 블록을 관리한다.
- 디렉토리는 파일 이름과 i-node 번호를 매핑하는 구조로 이루어져 있다.

특징:

1. 저널링 (Journaling) 기능 없음  
→ 파일 시스템 손상 시 복구 시간이 오래 걸릴 수 있음

*Journaling* : 파일 시스템에서 파일에 실제 쓰기 작업을 하기 전에 저널이라는 공간에 변경 사항을 저장
- 갑작스러운 오류나 시스템 충돌 등으로 데이터가 손상되는 것을 방지하여 데이터 무결성을 보장
- 시스템 오류가 발생하여도 저널을 통해 빠르게 복구 가능
- 

3. 효율적인 블록 관리  
→ 데이터 블록을 그룹 단위로 관리하여 접근 성능 향상  

4. 높은 호환성  
→ 구조가 단순하여 다양한 시스템에서 사용 가능

### 4. 개발 또는 일상과 연결해 본 예시

우리가 사용하는 파일 탐색기에서 폴더를 열고 파일을 찾는 과정은 디렉토리 구조를 기반으로 이루어진다. 파일을 실행하거나 저장할 때 운영체제가 해당 파일의 위치를 찾아 접근한다.

또한 대용량의 파일을 다룰 때는, 파일 시스템이 데이터를 블록 단위로 나누어 저장하고 관리하기 때문에 효율적인 읽기/쓰기가 가능하다.

리눅스를 포함한 유닉스 계열 운영체제에서는 i-node 기반으로 파일을 관리하기 때문에, 파일 이름과 실제 데이터 저장 위치가 분리되어 있다는 특징이 있다.

---

### **실제 개발이나 운영 상황에서는 어떤 trade-off가 있을까?**

파일 시스템은 데이터 접근 속도, 저장 효율성, 안정성 사이에서 trade-off가 존재한다.

예를 들어 연속 할당 방식은 접근 속도가 빠르지만 외부 단편화가 발생할 수 있고, 연결 할당은 외부 단편화를 줄일 수 있지만 임의 접근이 어렵다는 단점이 있다. 색인 할당은 임의 접근이 가능하지만 인덱스 블록을 위한 추가 공간이 필요하다.

또한 파일 시스템 구조에서도 단순한 구조는 구현이 쉽고 빠르지만 장애 발생 시 복구가 어려울 수 있고, 반대로 안정성을 높이기 위한 기능을 추가하면 구조가 복잡해지고 성능 오버헤드가 발생할 수 있다.
