## 전체 학습 소감

이번 프리코스를 통해 이미 알고 있던 운영체제 개념들을 단순히 암기 수준에서 벗어나, 전체 흐름 속에서 다시 정리할 수 있었다.

전공 수업을 통해 프로세스, 스케줄링, 동기화, 가상 메모리, 파일 시스템과 같은 개념들을 개별적으로 학습한 경험은 있었지만, 이번에는 각 개념들이 어떻게 연결되어 실제 시스템에서 하나의 구조로 동작하는지를 중심으로 이해하게 되었다. 

예를 들어 CPU 스케줄링은 단순한 알고리즘 선택 문제가 아니라, 프로세스 상태 변화와 컨텍스트 스위칭, 그리고 동기화 문제와까지 이어진다는 점을 다시 정리할 수 있었다.

또한 연속 메모리 할당에서 발생하는 단편화 문제를 시작으로 가상 메모리, 페이징, 페이지 교체로 이어지는 흐름이나, 동기화에서 경쟁 상태를 해결하는 과정이 잘못 설계될 경우 교착 상태로 이어질 수 있다는 점 등, 하나의 문제를 해결하는 과정에서 또 다른 문제가 발생하고 이를 다시 해결하는 구조를 이해하게 된 것이 인상적이었다.

AI를 활용한 학습 과정도 의미 있었다. 단순히 개념을 검색하는 수준을 넘어서, 헷갈리는 부분이나 부족한 내용을 질문하고 답변을 바탕으로 다시 정리하면서 이해를 보완할 수 있었다. 특히 Deadlock 해결 방식, 페이지 교체 알고리즘과 같은 부분은 AI를 통해 추가적으로 탐색하며 기존 지식을 확장할 수 있었다.

다만 실제 운영체제에서 이러한 개념들이 현업에서 어떻게 구현되고 적용되는지에 대해서는 여전히 부족함을 느꼈다. 특히 스케줄링 정책이나 메모리 관리 방식, 동기화 기법이 실제 서버 환경에서 요청 처리나 동시성 제어에 어떤 영향을 미치는지까지는 깊이 있게 이해하지 못한 부분이 있었다.

백엔드 개발자를 목표로 하고 있는 만큼, 이러한 운영체제 개념들을 단순한 이론으로 두는 것이 아니라, 실제 서버에서의 요청 처리 흐름, 멀티스레드 환경에서의 동시성 문제, 그리고 성능 최적화와 연결해서 이해하는 것이 중요하다고 생각하게 되었다.

이번 프리코스를 통해 기초 지식을 튼튼하게 한 것을 발판 삼아, 실제 백엔드 개발 과정에서의 문제 해결과 성능 개선으로 연결해 학습을 이어가고 싶다.

# 🚀 CH 10. 운영체제 시작하기


### 1. 핵심 개념 3개

- 운영체제와 커널 (OS & Kernel)
- 이중 모드 (User mode / Kernel mode)
- 시스템 콜과 프로세스

---

### 2. 내 말로 설명하기

운영체제는 프로그램이 실행될 때 필요한 CPU, 메모리, 입출력 장치 같은 자원을 관리하고, 프로그램이 정상적으로 실행될 수 있도록 도와주는 프로그램이다. 운영체제 역시 하나의 프로그램이기 때문에 메모리에 올라가 있어야 하며, 그 핵심 기능을 담당하는 부분이 커널이다.

운영체제는 시스템의 안전성을 위해 이중 모드라는 개념을 사용한다. 사용자 프로그램은 User mode에서 실행되며, 이 상태에서는 CPU나 메모리 같은 중요한 자원에 직접 접근할 수 없다. 반면 Kernel mode에서는 모든 자원에 접근할 수 있으며, 운영체제가 이 모드에서 동작한다.

사용자 프로그램이 자원이 필요할 때는 시스템 콜을 통해 운영체제에 요청을 보내고, 이 과정에서 User mode에서 Kernel mode로 전환된다. 이 과정을 통해 운영체제가 직접 자원을 관리하고, 프로그램은 안전하게 실행될 수 있다.

프로세스는 이러한 자원을 할당받아 실제로 실행 중인 프로그램을 의미하며, 운영체제는 여러 프로세스를 동시에 관리하면서 CPU를 효율적으로 분배한다.

---

### 3. 추가로 찾아본 내용 (AI 이용)

**Flag Register**  
- CPU가 연산 결과나 상태를 기록해두는 상태 레지스터  
    (ex, 연산 결과가 0인지, 음수인지, 오버플로우가 발생했는지)  
  
1. Zero Flag (ZF) - 연산 결과가 0이면 1
2. Sign Flag (SF) - 연산 결과가 음수이면 1
3. Carry Flag (CF) - 2진수 기준 자리 올림이 발생하면 1 (unsigned 연산에서 중요)
4. Overflow Flag (OF) - 표현 범위를 초과한 경우 1 (signed 연산에서 중요)
5. Interrupt Flag (IF) - 인터럽트 허용하는 경우 1 (OS가 제어하는 flag)
6. Supervisor Flag - 현재 CPU가 Kernel mode 이면 1, User mode 이면 0  
  
**실제 흐름 예시 (시스템 콜 발생 시)**  
1. User Mode에서 실행 중  
2. 시스템 콜 호출 (trap)  
3. CPU가 Supervisor bit = 1로 변경  
4. Kernel Mode 진입  
5. 작업 수행  
6. 반환 값과 함께 User Mode 복귀  

---

**Interrupt**
- 외부 장치에 의해 발생하는 비동기 이벤트
    (ex, 키보드 입력, 디스크 작업 완료)
  
→ 현재 실행 중인 작업을 잠시 멈추고 인터럽트 처리 후 다시 복귀  

---

**Trap**  
- 프로그램 실행 중 내부에서 발생하는 동기 이벤트  
    (ex, 시스템 콜, 예외 상황)

→ 사용자 프로그램이 운영체제 기능을 요청할 때 발생  

---

**Interrupt vs Trap 차이**

- Interrupt → 외부, 비동기  
- Trap → 내부, 동기  

---

**Context Switching**  
- CPU가 한 프로세스에서 다른 프로세스로 전환되는 과정  

→ 기존 프로세스 상태 저장 + 새로운 프로세스 상태 복원  

(저장되는 정보: 레지스터 값, PC(프로그램 카운터) 등)  
  
**Context Switching 실제 흐름 예시 (스케줄링)**  
  
1. 프로세스 A 실행 중  
2. 타이머 인터럽트 발생  
3. CPU가 Kernel Mode로 전환  
4. 현재 프로세스 A 상태 저장  
5. 스케줄러가 다음 프로세스 B 선택  
6. 프로세스 B 상태 복원  
7. 프로세스 B 실행

---

### 4. 개발 또는 일상과 연결해 본 예시

사용자가 파일을 읽거나 네트워크 요청을 보낼 때, 프로그램이 직접 하드웨어에 접근하는 것이 아니라 운영체제에 요청을 보내서 처리된다. 예를 들어 서버에서 클라이언트의 요청을 처리할 때도 내부적으로는 시스템 콜을 통해 네트워크 입출력이 이루어진다.

또한 xv6 프로젝트에서 시스템 콜을 구현하면서, 사용자 프로그램이 커널에 요청을 보내고 커널이 이를 처리한 뒤 다시 사용자 영역으로 돌아오는 흐름을 직접 확인할 수 있었다.

---

## **이 개념은 왜 필요한가?**


운영체제가 없다면 각 프로그램이 CPU나 메모리, 입출력 장치를 직접 제어해야 하기 때문에 시스템이 매우 복잡해지고 안정성이 크게 떨어질 것이다. 특히 여러 프로그램이 동시에 실행되는 환경에서는 자원 충돌이나 데이터 손상이 쉽게 발생할 수 있다.

이중 모드와 시스템 콜 구조는 이러한 문제를 해결하기 위해 만들어진 것으로, 사용자 프로그램이 직접 자원에 접근하지 못하게 제한하고, 운영체제를 통해서만 접근하도록 하여 안정성과 보안을 보장한다.

다만 이러한 구조는 시스템 콜이나 Context Switching 과정에서 **성능 비용**이 발생한다는 trade-off가 존재한다.

# 🚀 CH 11. 프로세스와 스레드  

### 1. 핵심 개념 3개  

- 프로세스
- 스레드
- 멀티 프로세스 vs 멀티 스레드

---

### 2. 내 말로 설명하기  

**프로세스**는 자원을 할당받아 실행 중인 프로그램을 의미하며, CPU를 할당받아야 실제로 실행될 수 있다. 여러 개의 프로세스가 동시에 실행되는 것처럼 보이지만 실제, CPU를 번갈아 사용하면서 동시에 실행되는 것처럼 보인다.

운영체제는 각 프로세스를 관리하기 위해 **PCB(Process Control Block)**라는 자료구조를 사용하며, 여기에는 PID, 레지스터 값, 상태, 메모리 정보 등이 저장된다. 특히 컨텍스트 스위칭 시 이전 상태를 복원하기 위해 레지스터 값이 중요하다.

프로세스는 생성, 준비, 실행, 대기, 종료 상태를 가지며, 상황에 따라 상태가 전환된다. (ex, 실행 중인 프로세스는 타이머 인터럽트가 발생하면 준비 상태로 돌아가고, I/O 작업이 발생하면 대기 상태로 전환된다.)  

**스레드**는 프로세스 내부의 실행 흐름 단위로, 하나의 프로세스 내에서 여러 스레드가 동시에 실행될 수 있다. 스레드는 코드, 데이터, 힙 영역을 공유하고, 각자 프로그램 카운터와 스택을 가진다.  

**멀티 프로세스**는 여러 개의 프로세스를 실행하는 방식으로 각 프로세스가 독립된 메모리 공간을 가지기 때문에 안정성이 높다. 반면 **멀티 스레드**는 하나의 프로세스 내에서 여러 스레드를 실행하는 방식으로 자원을 공유하기 때문에 효율적이지만 동기화 문제가 발생할 수 있다.

---

### 3. 추가로 찾아본 내용 (AI 활용)  


**스레드 동기화 문제 해결 방법**  

- 여러 스레드가 공유 자원에 동시에 접근하면 데이터 불일치 문제가 발생할 수 있다.  

**1. Mutex (Mutual Exclusion) - 상호배제**
- 하나의 스레드만 임계 구역에 접근할 수 있도록 lock을 거는 방식  
- 다른 스레드는 lock이 해제될 때까지 대기


**2. Semaphore**
- 여러 개의 자원에 대해 접근을 제어하는 방식  
- 카운팅 값을 통해 동시에 접근 가능한 스레드 수를 제한

→ 이를 통해 멀티 스레드 환경에서 발생하는 경쟁 상태(race condition)를 방지할 수 있다.  

**경쟁 상태(race condition)** : 여러 프로세스 또는 스레드가 동시에 같은 자원에 접근하여 결과가 실행 순서에 따라 달라지는 상황

---

**IPC (Inter-Process Communication)**

- 서로 다른 프로세스 간에 데이터를 주고받기 위한 방법

**1. Pipe**  
- 한 프로세스에서 다른 프로세스로 데이터를 한 방향으로 전달  

**2. Message Queue**  
- 메시지를 큐 형태로 저장하고, 프로세스가 이를 읽는 방식  

**3. Shared Memory**  
- 분리된 여러 프로세스가 일부 메모리 공간을 의도적으로 공유하여 빠르게 데이터 교환 (스레드의 장점을 적용하여 사용)  
- 대신 동기화 문제가 발생할 수 있음  

**4. Socket**  
- 네트워크를 통해 다른 프로세스와 통신하는 방식  
- 서버-클라이언트 구조에서 사용됨  

→ IPC를 통해 독립적인 프로세스 간에도 협력하여 작업을 수행할 수 있다.  

---

### 4. 개발 또는 일상과 연결해 본 예시


파일을 다운로드하면서 동시에 다른 작업을 수행할 수 있는 이유는 다운로드 작업이 별도의 스레드 또는 프로세스로 처리되기 때문이다.

이 과정에서 CPU는 여러 작업을 번갈아 수행하며 동시에 실행되는 것처럼 보이게 한다.  

웹 서버에서는 여러 사용자의 요청을 동시에 처리하기 위해 스레드 기반으로 요청을 분산 처리한다.

이때 스레드를 사용하면 자원을 효율적으로 사용할 수 있지만, 동시에 같은 데이터를 접근할 경우 동기화 문제가 발생할 수 있다는 점도 함께 고려해야 한다.

---

## 실제 개발이나 운영 상황에서는 어떤 trade-off가 있을까?

멀티 프로세스는 각 프로세스가 독립된 메모리 공간을 가지기 때문에 안정성이 높지만, 생성 비용과 메모리 사용량이 크다.

반면 멀티 스레드는 자원을 공유하기 때문에 효율적이지만, 동기화 문제가 발생할 수 있고 하나의 스레드에서 문제가 생기면 전체 프로세스에 영향을 줄 수 있다는 단점이 있다.

# 🚀 CH 12. CPU 스케줄링

### 1. 핵심 개념 3개

- CPU 스케줄링
- 선점형 vs 비선점형 스케줄링
- 스케줄링 알고리즘

---

### 2. 내 말로 설명하기

CPU 스케줄링은 운영체제가 여러 프로세스에게 CPU 자원을 공정하고 효율적으로 분배하는 과정이다. CPU는 한 번에 하나의 프로세스만 실행할 수 있기 때문에, 여러 프로세스를 번갈아 실행시키며 동시에 처리되는 것처럼 보이게 한다.

일반적으로 I/O bound 프로세스는 CPU를 짧게 사용하고 대기 상태에 자주 들어가기 때문에 높은 우선순위를 가지는 경우가 많고, CPU bound 프로세스는 상대적으로 낮은 우선순위를 가지게 된다. 이를 통해 전체 시스템의 응답성을 높일 수 있다.

스케줄링 방식은 크게 선점형과 비선점형으로 나뉜다. 선점형 스케줄링은 현재 실행 중인 프로세스의 CPU를 빼앗아 다른 프로세스에게 할당할 수 있는 방식으로, 공정성과 응답성이 높지만 컨텍스트 스위칭으로 인한 오버헤드가 발생한다. 반면 비선점형 스케줄링은 프로세스가 작업을 끝낼 때까지 CPU를 계속 사용하는 방식으로 오버헤드는 적지만 특정 프로세스가 CPU를 오래 점유할 수 있다는 단점이 있다.

대표적인 스케줄링 알고리즘으로는 FIFO, SJF, RR, Priority Scheduling, Multi-level Queue, Multi-level Feedback Queue 등이 있으며, 각각 공정성, 응답성, 구현 난이도 등의 측면에서 장단점이 존재한다. 특히 MLFQ는 프로세스의 실행 패턴에 따라 우선순위를 동적으로 조정하여 CPU bound와 I/O bound 프로세스를 효율적으로 구분할 수 있는 방식이다.

---

### 3. 추가로 찾아본 내용 (AI 활용)

**유닉스의 FSS (Fair Share Scheduler)**

- FSS는 사용자 또는 그룹 단위로 CPU 자원을 공정하게 분배하는 스케줄링 방식이다.
- 각 사용자(또는 그룹)에 CPU 사용 비율(share)을 할당하고, 해당 비율에 맞게 CPU 시간을 나누어 준다.
- 동일한 시스템에서 여러 사용자가 동시에 작업할 때 특정 사용자가 CPU를 독점하지 못하도록 하기 위한 방식이다.

예를 들어,  
사용자 A와 사용자 B가 각각 50%의 share를 가진다면,  
A가 여러 개의 프로세스를 실행하더라도 A의 전체 CPU 사용량은 50%를 넘지 않도록 제한된다.

**→ 즉, “프로세스 단위”가 아니라 “사용자 단위”로 공정성을 보장하는 방식이다.**

---

**리눅스의 CFS (Completely Fair Scheduler)**

- CFS는 모든 프로세스가 CPU를 공평하게 사용하도록 설계된 리눅스의 기본 스케줄러이다.
- 각 프로세스의 실행 시간을 가상 실행 시간(Virtual Runtime, vruntime)으로 관리한다.
- vruntime이 가장 작은 프로세스 즉, 가장 적게 CPU를 사용한 프로세스를 우선적으로 실행한다.

**→ 지금까지 덜 실행된 프로세스를 먼저 실행시켜 공정성을 맞춘다.**


**CFS의 주요 특징**

1. Red-Black Tree 사용  
- 프로세스를 가상 실행 시간(vruntime) 기준으로 정렬하여 빠르게 가장 작은 값을 찾을 수 있다.
  
  **Red-Black Tree란?**
  자가 균형 이진 탐색 트리라고 불리며, 이름과 같이 특정 규칙들을 통해 균형을 유지하는 특수한 이진 탐색 트리이다.

2. 선점형 스케줄링  
- vruntime 값이 더 작은 프로세스(우선순위가 더 높은 프로세스)가 있으면 현재 프로세스를 중단시키고 교체한다.

3. 우선순위 반영  
- nice 값에 따라 vruntime 증가 속도가 달라진다.  
- 우선순위가 높은 프로세스는 vruntime이 느리게 증가하여 더 자주 실행된다.

  **nice 값이란?**
  프로세스의 CPU 우선순위를 조정하는 값으로 -20 ~ 19의 범위를 가지고 있고, 값이 낮을 수록 높은 우선순위를 가진다.
  nice 값이 낮은 프로세스는 vruntime 값이 더 느리게 증가하여 더 자주 CPU를 할당 받을 수 있도록 한다.
  즉, nice 값은 CFS에서 프로세스 간 공정성 유지하면서, 우선순위를 조정할 수 있는 역할을 한다.

---

**FSS vs CFS 차이**

| 구분 | FSS (유닉스) | CFS (리눅스) |
|------|-------------|-------------|
| 기준 | 사용자 | 프로세스 |
| 목표 | 사용자 간 공정성 | 프로세스 간 공정성 |
| 방식 | 할당된 share 비율 기반 | vruntime 기반 |
| 특징 | 특정 사용자 독점 방지 | 가장 덜 실행된 프로세스 우선 |

→ FSS는 “사용자 단위 공정성”  
→ CFS는 “프로세스 단위 공정성”에 초점을 둔다는 차이가 있다.

### 4. 개발 또는 일상과 연결해 본 예시

웹 서버에서 여러 사용자의 요청이 동시에 들어올 때, CPU는 각 요청을 하나씩 처리하는 것이 아니라 스케줄링을 통해 빠르게 번갈아 실행한다. 이 과정에서 I/O 작업이 많은 요청은 빠르게 처리되고, CPU를 오래 사용하는 작업은 뒤로 밀리면서 전체적인 응답 속도가 향상된다.

또한 xv6 프로젝트에서 MLFQ 스케줄러를 구현하면서, 프로세스의 CPU 사용량에 따라 우선순위를 조정하고 큐 간 이동을 통해 공정성과 효율성을 동시에 고려하는 방식이 실제로 어떻게 동작하는지 경험할 수 있었다.

---

### **실제 개발이나 운영 상황에서는 어떤 trade-off가 있을까?**

CPU 스케줄링 알고리즘은 공정성, 응답성, 처리량 사이에서 trade-off가 존재한다. 예를 들어 RR은 공정성과 응답성은 좋지만 그만큼 컨텍스트 스위칭 비용이 증가할 수 있고, SJF는 평균 대기 시간을 줄일 수 있지만 실행 시간을 정확히 예측하기 어렵다.

또한 우선순위 기반 스케줄링은 중요한 작업을 빠르게 처리할 수 있지만, 낮은 우선순위 프로세스가 계속 실행되지 못하는 starvation 문제가 발생할 수 있다. 이러한 문제점을 해결하기 위해 Aging과 같은 기법이 사용되며, 실제 시스템에서는 이러한 다양한 요소를 고려하여 효율적으로 스케줄링 정책이 설계된다.

# 🚀 CH 13. 프로세스 동기화

### 1. 핵심 개념 3개

- 동기화 (Synchronization)
- 임계 구역과 경쟁 상태 (Critical Section & Race Condition)
- 동기화 기법 (Mutex, Semaphore, Monitor)

---

### 2. 내 말로 설명하기

**동기화**는 여러 프로세스나 스레드가 동시에 실행될 때, 실행 순서를 맞추고 공유 자원의 일관성을 유지하기 위한 메커니즘이다. 여러 작업이 협력하여 하나의 결과를 만들어내는 경우, 순서가 맞지 않거나 동시에 접근하게 되면 결과 값이 달라지거나 데이터가 깨질 수 있기 때문에 동기화는 필수이다.

이 과정에서 중요한 개념이 **임계 구역**과 **경쟁 상태**이다. 임계 구역은 동시에 실행되면 문제가 발생할 수 있는 코드 영역을 의미하며, 공유 자원에 접근하는 부분이 해당된다. 여러 프로세스가 동시에 임계 구역에 접근하면 데이터 불일치가 발생할 수 있는데, 이를 경쟁 상태라고 한다.

이 문제를 해결하기 위해 운영체제는 상호 배제 (mutal exclusion), 진행 (progress), 유한 대기 (bounded waiting) 이라는 조건을 만족해야 한다. 쉽게 말해, 한 번에 하나의 프로세스만 접근할 수 있어야 하고, 아무도 사용하지 않는다면 접근할 수 있어야 하며, 무한히 기다리는 상황이 없어야 한다.

이를 구현하기 위한 대표적인 동기화 기법으로는 Mutex, Semaphore, Monitor가 있다. Mutex는 lock을 통해 한 번에 하나의 프로세스만 접근하도록 제어하고, Semaphore는 동시에 접근 가능한 개수를 제어한다. Monitor는 이러한 동기화를 더 쉽게 사용할 수 있도록 추상화한 구조로, 모니터에 하나의 프로세스만 접근할 수 있도록 보장한다.

---

### 3. 추가로 찾아본 내용 (AI 활용)

**Deadlock (교착 상태)**

- Deadlock은 여러 프로세스가 서로가 가진 자원을 기다리면서 무한히 대기하는 상태이다.

예를 들어,  
프로세스 A는 자원 1을 가지고 자원 2를 기다리고 있고,  
프로세스 B는 자원 2를 가지고 자원 1을 기다리는 경우  
→ 두 프로세스 모두 더 이상 진행할 수 없는 상태가 되고 이를 교착 상태 (Deadlock) 이라고 한다.

---

**Deadlock 발생 조건 (4가지)**

1. 상호 배제 (Mutual Exclusion)  
- 한 번에 하나의 프로세스만 자원을 사용할 수 있음  

2. 점유와 대기 (Hold and Wait)  
- 자원을 하나 이상 보유한 상태에서 다른 자원을 기다림  

3. 비선점 (No Preemption)  
- 이미 할당된 자원을 강제로 빼앗을 수 없음  

4. 순환 대기 (Circular Wait)  
- 프로세스들이 원형 구조로 서로 자원을 기다림  

→ 위 4가지 조건이 모두 만족되면 Deadlock이 발생한다.

---

**Deadlock 해결 방법**

1. 예방 (Prevention)  
- 4가지 조건 중 하나를 제거하여 Deadlock 자체를 발생하지 않도록 함  
- 예: 자원을 한 번에 모두 요청하도록 하여 Hold and Wait 제거  

2. 회피 (Avoidance)  
- 시스템이 안전 상태(Safe State)를 유지하도록 자원 할당  
- 대표적으로 은행원 알고리즘이 사용됨  

3. 탐지 및 회복 (Detection & Recovery)  
- Deadlock 발생을 허용하고, 이후 탐지하여 해결  
- 예: 특정 프로세스 종료 또는 자원 강제 회수  

---

**Deadlock vs Race Condition 차이**

- Race Condition: 여러 프로세스가 동시에 자원에 접근하여 데이터가 꼬이는 문제  
- Deadlock: 서로 자원을 기다리며 무한히 멈춰버리는 문제  

→ 동기화는 Race Condition을 해결하지만, 잘못 설계하면 Deadlock을 유발할 수 있다.

---

### 4. 개발 또는 일상과 연결해 본 예시

멀티 스레드 환경에서 여러 스레드가 동시에 하나의 전역 변수에 접근하면 값이 꼬일 수 있는데, 이를 방지하기 위해 Mutex나 Semaphore를 사용하여 접근을 제한한다.

또한 두 개의 스레드가 서로 다른 자원을 점유한 상태에서 서로의 자원을 기다리는 경우 Deadlock이 발생할 수 있다. 예를 들어, 하나의 스레드는 파일을 잠그고 DB 접근을 기다리고 있고, 다른 스레드는 DB를 점유한 상태에서 파일 접근을 기다리는 경우 시스템이 멈출 수 있다.

---

### **실제 개발이나 운영 상황에서는 어떤 trade-off가 있을까?**

동기화를 통해 데이터 일관성을 보장할 수 있지만, 동시에 성능 저하가 발생할 수 있다. 

(ex. lock을 많이 사용할수록 병렬성이 줄어들어 성능이 떨어질 수 있다.)

또한 잘못된 동기화 설계는 Deadlock이나 starvation과 같은 문제를 유발할 수 있다. 

따라서 실제 시스템에서는 성능과 안정성 사이의 균형을 고려하여 적절한 동기화 기법을 선택하는 것이 중요하다.

# 🚀 CH 14. 교착 상태 (Deadlock)

### 1. 핵심 개념 3개

- 교착 상태 (Deadlock)
- 교착 상태 발생 조건 (4가지)
- 교착 상태 해결 방법

---

### 2. 내 말로 설명하기

교착 상태는 여러 프로세스들이 서로가 가진 자원을 기다리면서 더 이상 진행하지 못하는 상태를 의미한다. 즉, 앞으로 일어나지 않을 사건을 기다리며 무한히 멈춰 있는 상태이다.

자원 할당 그래프 교착 상태의 조건을 파악하기 위해 사용할 수 있다. 이 그래프에서 원형 구조(Circular wait)가 발생하면 교착 상태가 발생할 가능성이 있다.

교착 상태는 네 가지 조건이 동시에 만족될 때 발생한다. 상호 배제 (mutal exclusion), 점유와 대기 (hold and wait), 비선점 (non preemption), 원형 대기 (circular wait) 조건이 모두 충족되면 프로세스들이 서로 자원을 기다리며 멈추게 된다.

이를 해결하기 위해  예방, 회피, 검출 후 회복같은 방법이 존재한다. 예방은 아예 발생하지 않도록 조건을 제거하는 방식이고, 회피는 안전 상태를 유지하도록 자원을 할당하는 방식이다. 검출 후 회복은 교착 상태를 허용한 뒤 발생하면 해결하는 방식이다.

---

### 3. 추가로 찾아본 내용 (AI 활용)

**은행원 알고리즘 (Banker's Algorithm)**

- 시스템이 항상 안전 상태(Safe State)를 유지하도록 자원을 할당하는 방식
- 프로세스가 자원을 요청하면, 이를 미리 가상으로 할당해보고 안전 상태인지 검사한 후 실제 할당 여부를 결정

1. 안전 상태 (Safe State)  
- 모든 프로세스가 정상적으로 자원을 할당받고 종료될 수 있는 상태  

2. 불안전 상태 (Unsafe State)  
- 교착 상태가 발생할 가능성이 있는 상태  

3. 안전 순서열 (Safe Sequence)  
- 모든 프로세스가 순서대로 자원을 할당받아 종료될 수 있는 순서  

→ 고정된 수의 프로세스와 자원이 존재한다고 가정하며, 각 프로세스들이 사용할 요구량을 미리 운영체제가 알아야 한다는 단점이 있다.

---

### 4. 개발 또는 일상과 연결해 본 예시

멀티 스레드 환경에서 두 개의 스레드가 서로 다른 자원을 점유한 상태에서 상대방이 가진 자원을 기다리는 경우 교착 상태가 발생할 수 있다.

간단한  예시를 하나 들어, 하나의 스레드는 파일을 점유한 상태에서 DB 접근을 기다리고 있고, 다른 스레드는 DB를 점유한 상태에서 파일 접근을 기다리는 경우 두 스레드는 교착 상태에 빠진다.

이처럼 실제 서버 환경에서도 자원 접근 순서를 잘못 설계하면 교착 상태 발생 가능성이 존재하기 때문에, 자원 접근 순서를 통일하거나 lock을 사용하여 이를 예방해야 한다.

---

### **실제 개발이나 운영 상황에서는 어떤 trade-off가 있을까?**

교착 상태를 예방하기 위해 자원 요청을 제한하거나 순서를 강제로 정하면 시스템 자원의 활용도가 떨어질 수 있다. 

반대로 교착 상태를 허용하고 탐지 및 회복 방식으로 처리하면 시스템의 유연성은 높아지지만, 문제가 발생했을 때 프로세스를 강제로 종료해야 하는 위험이 존재한다.

따라서 실제 시스템에서는 안정성과 자원 활용 효율성 사이에서 trade-off를 고려하여 적절한 방식으로 교착 상태를 관리해야 한다.

# 🚀 CH 15. 가상 메모리

### 1. 핵심 개념 8개

- 연속 메모리 할당
- 스와핑
- 가상 메모리
- 페이징과 페이지 테이블, 페이지 테이블 엔트리
- 쓰기 시 복사 (Copy on Write)
- 페이지 교체
- 스래싱 (Thrashing)
- 프레임 할당

---

### 2. 내 말로 설명하기

연속 메모리 할당은 프로세스에게 하나의 연속된 메모리 공간을 할당하는 방식이다. 구현이 단순하지만 *외부 단편화*가 발생한다는 단점을 가지고 있고, 물리 메모리보다 큰 프로세스를 실행할 수 없다는 한계가 있다.

이를 보완하기 위해 스와핑이 사용된다. 사용되지 않는 프로세스를 보조기억장치로 옮기고, 그 자리에 사용될 프로세스를 적재하는 방식이다. 이를 통해 물리 메모리보다 더 큰 프로그램도 실행할 수 있다.

*가상 메모리*는 이러한 개념을 확장한 기술로, 프로그램 전체를 메모리에 로드하지 않고 필요한 일부분만 적재하여 실행한다. 따라서 물리 메모리보다 큰 프로그램을 효율적으로 실행할 수 있고,

이 과정에서 사용되는 대표적인 기법이 페이징 기법이다. *페이징 기법*은 프로세스를 페이지라고 하는 단위로 나누고, 물리 메모리를 프레임 단위로 나누어 매핑하는 방식이다. 외부 단편화 문제를 개선할 수는 있지만, 내부 단편화가 발생할 수 있다.

*페이지 테이블*은 논리 주소를 물리 주소로 변환하기 위한 구조로써, 각 페이지가 어떤 프레임에 저장되어 있는지를 관리한다. 페이지 테이블 엔트리에는 유효 비트, 보호 비트, 참조 비트, 수정 비트 등이 포함되어 페이지 상태를 관리한다.

쓰기 시 복사(Copy on Write)는 프로세스 복제 시 메모리를 바로 복사하지 않고, 메모리 관리와 최적화를 위해 실제로 쓰기 작업이 발생할 때만 복사하는 방식이다.

요구 페이징 환경에서는 메모리가 가득 찰 경우 어떤 페이지를 제거할지 결정해야 하는데, 이를 페이지 교체라고 하며, FIFO, LRU, Optimal 알고리즘이 존재한다.

또한 프레임을 어떻게 할당하느냐에 따라 성능이 달라지며, 잘못된 할당은 페이지 교체가 너무 자주 발생하여 실제 실행보다 페이지 교체에 더 많은 시간이 소요되어 성능이 저해되는 스래싱을 유발할 수 있다. 

---

### 3. 추가로 찾아본 내용 (AI 활용)

**세그멘테이션 (Segmentation)**

- 세그멘테이션은 프로그램을 의미 단위(코드, 데이터, 스택, 힙)로 나누어 메모리를 할당하는 방식이다.
- 각 세그먼트는 크기가 다르며, 논리적으로 구분된 구조를 그대로 반영할 수 있다.
- 하지만 외부 단편화 문제가 발생할 수 있어 페이징과 함께 사용되는 경우가 많다.

---

**Clock 알고리즘**

- LRU 알고리즘을 근사적으로 구현한 페이지 교체 방식
- 원형 큐 형태로 페이지를 관리하며, 참조 비트(reference bit)를 사용

동작 방식:
1. 포인터가 가리키는 페이지의 참조 비트 확인
2. 참조 비트가 0이면 교체
3. 참조 비트가 1이면 0으로 변경 후 다음 페이지로 이동

→ 최근에 사용된 페이지는 한 번 더 기회를 얻는 방식이다.

---

**2-Hand Clock 알고리즘**

- Clock 알고리즘을 개선한 방식
- 두 개의 포인터를 사용하여 페이지 교체를 더 효율적으로 수행

→ 첫 번째 포인터(Front hand)는 참조 비트를 초기화하고,  
→ 두 번째 포인터(Back hand)는 실제 교체할 페이지(참조 비트가 0인 페이지)를 선택한다.

→ 이를 통해 페이지 교체 판단을 더 정확하게 할 수 있다.

---

### 4. 개발 또는 일상과 연결해 본 예시

여러 개의 프로그램을 동시에 실행할 때, 모든 프로그램이 메모리에 올라가 있는 것이 아니라 필요한 부분만 메모리에 올라가고 나머지는 디스크에 저장된다. 이는 가상 메모리와 요구 페이징 덕분에 가능한 동작이다.

만약 실행 중인 프로그램이 너무 많아 메모리가 부족해지면, 페이지 교체가 빈번하게 발생하게 되고 이로 인해 성능이 급격히 저하되는 스래싱 현상이 발생할 수 있다.

---

### 5. **실제 개발이나 운영 상황에서는 어떤 trade-off가 있을까?**

가상 메모리를 사용하면 물리 메모리보다 큰 프로그램을 실행할 수 있다는 장점이 있지만, 페이지 테이블 접근과 페이지 교체 과정에서 성능 오버헤드가 발생한다.

또한 페이지 교체 알고리즘을 잘못 선택하거나 프레임 할당이 적절하지 않을 경우 스래싱이 발생하여 시스템 성능이 크게 저하될 수 있다.

따라서 실제 운영체제에서는 메모리 사용 효율성과 성능 사이의 trade-off를 고려하여 다양한 기법을 조합하여 사용한다.

# 🚀 CH 16. 파일 시스템

### 1. 핵심 개념 3개

- 파일과 디렉토리
- 파일 할당 방법
- 파일 시스템 구조 (FAT, i-node)

---

### 2. 내 말로 설명하기

*파일 시스템*은 운영체제에서 파일과 디렉토리를 관리하는 프로그램으로, 데이터를 체계적으로 저장하고 접근할 수 있도록 도와준다. *파일*은 보조기억장치에 저장된 의미 있는 정보의 집합이며, 반드시 운영체제를 통해서만 접근할 수 있다.

*디렉토리*는 파일을 효율적으로 관리하기 위한 구조로, 파일의 위치와 정보를 저장한다. 경로를 통해 특정 파일의 위치를 지정할 수 있으며, 절대 경로와 상대 경로가 존재한다.

파일은 보조기억장치에 *블록 단위*로 저장되며, 이를 어떻게 할당하느냐에 따라 성능과 효율이 달라진다.

연속 할당은 파일을 연속된 블록에 저장하는 방식으로 구현이 간단하지만 외부 단편화가 발생할 수 있다. 이를 해결하기 위해 불연속 할당 방식이 사용된다.

불연속 할당 방식 중 *연결 할당*은 각 블록이 다음 블록을 가리키는 구조로 외부 단편화를 줄일 수 있지만, 순차 접근만 가능하여 임의의 블록에 접근하는데 속도가 느리다는 단점이 있다. 반면 *색인 할당*은 모든 블록 주소를 인덱스 블록에 저장하여 임의 접근이 더 빠르다.

FAT 파일 시스템은 연결 정보를 테이블 형태로 관리하여 접근 속도를 개선한 방식이고, 유닉스 파일 시스템은 **i-node** 구조를 사용하여 파일의 블록 주소를 효율적으로 관리한다.

---

### 3. 추가로 찾아본 내용 (AI 활용)

**ext2 파일 시스템**

- ext2는 리눅스에서 사용되는 전통적인 파일 시스템 중 하나이다.
- i-node 기반 구조를 사용하여 파일의 메타데이터와 데이터 블록을 관리한다.
- 디렉토리는 파일 이름과 i-node 번호를 매핑하는 구조로 이루어져 있다.

특징:

1. 저널링 (Journaling) 기능 없음  
→ 파일 시스템 손상 시 복구 시간이 오래 걸릴 수 있음

*Journaling* : 파일 시스템에서 파일에 실제 쓰기 작업을 하기 전에 저널이라는 공간에 변경 사항을 저장
- 갑작스러운 오류나 시스템 충돌 등으로 데이터가 손상되는 것을 방지하여 데이터 무결성을 보장
- 시스템 오류가 발생하여도 저널을 통해 빠르게 복구 가능
- 

3. 효율적인 블록 관리  
→ 데이터 블록을 그룹 단위로 관리하여 접근 성능 향상  

4. 높은 호환성  
→ 구조가 단순하여 다양한 시스템에서 사용 가능  

---

### 4. 개발 또는 일상과 연결해 본 예시

우리가 사용하는 파일 탐색기에서 폴더를 열고 파일을 찾는 과정은 디렉토리 구조를 기반으로 이루어진다. 파일을 실행하거나 저장할 때 운영체제가 해당 파일의 위치를 찾아 접근한다.

또한 대용량의 파일을 다룰 때는, 파일 시스템이 데이터를 블록 단위로 나누어 저장하고 관리하기 때문에 효율적인 읽기/쓰기가 가능하다.

리눅스를 포함한 유닉스 계열 운영체제에서는 i-node 기반으로 파일을 관리하기 때문에, 파일 이름과 실제 데이터 저장 위치가 분리되어 있다는 특징이 있다.

---

### **실제 개발이나 운영 상황에서는 어떤 trade-off가 있을까?**

파일 시스템은 데이터 접근 속도, 저장 효율성, 안정성 사이에서 trade-off가 존재한다.

예를 들어 연속 할당 방식은 접근 속도가 빠르지만 외부 단편화가 발생할 수 있고, 연결 할당은 외부 단편화를 줄일 수 있지만 임의 접근이 어렵다는 단점이 있다. 색인 할당은 임의 접근이 가능하지만 인덱스 블록을 위한 추가 공간이 필요하다.

또한 파일 시스템 구조에서도 단순한 구조는 구현이 쉽고 빠르지만 장애 발생 시 복구가 어려울 수 있고, 반대로 안정성을 높이기 위한 기능을 추가하면 구조가 복잡해지고 성능 오버헤드가 발생할 수 있다.
